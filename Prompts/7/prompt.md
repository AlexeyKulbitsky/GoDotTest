Take the existing project as your starting point. There are four things I want you to work on in this task: a checkpoint system, a level transition mechanism, a second level, and a small but important refactor to the existing scene naming. Let me go through each one.

First, rename platformer.tscn to level_1.tscn and update every reference to it across the project — this includes the main scene path in Project Settings, any scene load calls in menu.gd, and anywhere else the filename is referenced. The game should work exactly as before after this rename with no broken references.

Second, introduce a checkpoint system. A checkpoint is a static marker in the level that the player activates by touching it. Save it as checkpoint.tscn with its script checkpoint.gd. Each checkpoint should have an Area2D for detection and a simple visual — a colored shape is fine. When the player touches an inactive checkpoint it becomes active, changes its color to give clear visual feedback that it has been activated, and stores itself as the current respawn position. Only one checkpoint can be active at a time — activating a new one should deactivate the previous one and revert it to its inactive color. When the player dies, they respawn at the last activated checkpoint instead of the original spawn point. If no checkpoint has been activated yet the player still respawns at the original spawn point as before. Place two or three checkpoints in level_1.tscn at meaningful positions throughout the level.

Third, add a level transition. Create a goal area saved as level_exit.tscn with its script level_exit.gd. It should be a clearly visible Area2D placed at the end of the level. When the player touches it, the game loads level_2.tscn. The current score must carry across the transition — the score the player had at the end of level 1 should be the starting score in level 2. To achieve this, introduce a new autoload singleton saved as game_manager.gd and registered in Project Settings under AutoLoad. The game manager is responsible for holding the score so it persists across scene changes. All score increments that currently happen in platformer.gd should be rerouted through the game manager, and the UI should read its score value from there as well. The menu's Restart button should also reset the score in the game manager when starting a fresh session.

Fourth, create level_2.tscn as a new small but complete level. It should reuse the existing platform, enemy, collectible, checkpoint, and power-up scenes — no new scene types are needed. It should have its own spawn point, at least two checkpoints, a level exit that could point back to level 1 or simply print a completion message for now, a handful of collectibles, and at least one enemy of each type. The layout should be different enough from level 1 to feel like a new environment, but it does not need to be elaborate — a compact but playable arrangement is enough.

The game manager should be the single source of truth for the score for the entire game session. It should expose a method for incrementing the score, a method for resetting it, and a readable property for the current value. No other script should hold its own local score variable — everything goes through the game manager.
